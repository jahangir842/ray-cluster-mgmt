# vLLM on a Single Node

Serves a large language model on **one GPU** using vLLM's default local
executor — no Ray cluster required.

Supported models (all on **pc3-4500**):

| Model | Path | VRAM (float16) |
|-------|------|----------------|
| Llama 3.1-8B | `models/3.1-8b-instruct` | ~15 GB |
| Qwen3.5-9B | `models/Qwen3.5-9B` | ~18 GB |

---

## Prerequisites

- Conda environment activated: `conda activate ray-env`
- Model downloaded to `~/projects/vllm-deployment/vllm/models/`
- Port 8000 free (see Troubleshooting if not)
- `outlines` installed: `pip install outlines`

---

## Memory budget

### Llama 3.1-8B

In `float16` on a 24 GB GPU (RTX 4500 Ada / RTX 3090):

| Item | Size |
|------|------|
| Model weights (float16) | ~15.0 GB |
| Activation / overhead | ~0.3 GB |
| **KV cache (remainder)** | **~6.3 GB** |
| Total reserved at `--gpu-memory-utilization 0.9` | 21.6 GB |

### Qwen3.5-9B

In `float16` on a 24 GB GPU:

| Item | Size |
|------|------|
| Model weights (float16) | ~18.0 GB |
| Activation / overhead | ~0.3 GB |
| **KV cache (remainder)** | **~3.3 GB** |
| Total reserved at `--gpu-memory-utilization 0.9` | 21.6 GB |

KV cache is tighter for Qwen3.5-9B — use `--gpu-memory-utilization 0.95` to
squeeze out ~4.6 GB for KV cache when running at 8192 context length.

`--gpu-memory-utilization 0.4` only reserves 9.6 GB — less than either model's
weights — so vLLM will fail immediately with a KV cache allocation error.

## Step 1 — Launch vLLM

Run on **pc3-4500** (or any single node with sufficient VRAM):

### Llama 3.1-8B

```bash
conda activate ray-env

vllm serve ~/projects/vllm-deployment/vllm/models/3.1-8b-instruct \
  --dtype float16 \
  --gpu-memory-utilization 0.9 \
  --max-model-len 8192 \
  --enforce-eager \
  --host 0.0.0.0 \
  --port 8000 \
  --guided-decoding-backend outlines
```

### Qwen3.5-9B

```bash
conda activate ray-env

vllm serve ~/projects/vllm-deployment/vllm/models/Qwen3.5-9B \
  --dtype float16 \
  --gpu-memory-utilization 0.95 \
  --max-model-len 8192 \
  --enforce-eager \
  --host 0.0.0.0 \
  --port 8000 \
  --guided-decoding-backend outlines
```

Server is ready when you see:

```
INFO: Uvicorn running on http://0.0.0.0:8000 (Press CTRL+C to quit)
```

### Flag notes

| Flag | Value | Why |
|------|-------|-----|
| `--gpu-memory-utilization` | `0.9` (Llama) / `0.95` (Qwen) | Reserves that fraction of VRAM; Qwen3.5-9B needs 0.95 to leave enough room for KV cache at 8192 context |
| `--enforce-eager` | — | Skips CUDA graph capture, reducing peak memory during startup |
| `--max-model-len` | `8192` | Minimum context window for production use; both models support 8192 on a 24 GB GPU with the utilization values above |
| `--guided-decoding-backend` | `outlines` | Bypasses xgrammar, which crashes the server on JSON-mode requests (`response_format: json_object/json_schema`) due to a missing `TokenizerInfo.from_huggingface` method in the installed xgrammar version. `outlines` is a drop-in replacement with no impact on output quality. See `docs/troubleshooting-vllm.md` for full details. |

## Troubleshooting

### `ValueError: No available memory for the cache blocks`

`--gpu-memory-utilization` is too low. Check the startup log for the actual
numbers:

```
model weights take X GiB; ... the rest of the memory reserved for KV Cache is -Y GiB.
```

A negative KV cache value means the weights alone exceed the reserved budget.
Increase `--gpu-memory-utilization` (max `0.95`) — do not lower `--max-model-len` below 8192.

### Port 8000 already in use

```bash
sudo fuser -k 8000/tcp
```

### CUDA out of memory during model load

The GPU does not have enough free VRAM. Check for other processes holding the
GPU before launching:

```bash
nvidia-smi
```

Kill any stale vLLM or Python processes, then retry.
