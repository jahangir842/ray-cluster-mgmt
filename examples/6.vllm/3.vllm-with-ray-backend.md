# vLLM with Ray Backend — Setup Guide

Serves a large language model across **8 GPUs on multiple nodes** using vLLM
pipeline parallelism with Ray as the distributed executor.

---

## Step 1 — Set environment variables in `~/.bashrc`

Each node type has a different physical NIC. Add the correct block to `~/.bashrc`
on each node, then `source ~/.bashrc`.

> **GLOO_SOCKET_IFNAME:** Every node also has Docker bridge interfaces (`docker0`,
> `br-*`) in 172.x.x.x ranges. Without this var set in the Ray worker process,
> Gloo auto-detects and may pick one of those bridges — causing `connectFullMesh`
> failures.

> **VLLM_HOST_IP:** Set this **only on the head node** (where `vllm serve` runs).
> If set on worker nodes, every worker reports the head's IP as its own —
> vLLM detects duplicate IPs and refuses to start with:
> `RuntimeError: Every node should have a unique IP address`.
> After removing it from `.bashrc`, also run `unset VLLM_HOST_IP` in the current
> shell before restarting Ray — `source ~/.bashrc` does not unset already-exported
> variables.

**RTX 4500 worker nodes — pc1, pc2, pc4, pc5 (192.168.3.71–72, 74–75)**
```bash
export GLOO_SOCKET_IFNAME=enp0s31f6
export NCCL_SOCKET_IFNAME=enp0s31f6
export NCCL_IB_DISABLE=1
```

**RTX 3090 worker nodes — pc6, pc7, pc8 (192.168.3.76–78)**
```bash
export GLOO_SOCKET_IFNAME=eno1
export NCCL_SOCKET_IFNAME=eno1
export NCCL_IB_DISABLE=1
```

**Head node only — pc3-4500 (192.168.3.73)**
```bash
export GLOO_SOCKET_IFNAME=enp0s31f6
export NCCL_SOCKET_IFNAME=enp0s31f6
export NCCL_IB_DISABLE=1
export VLLM_HOST_IP=192.168.3.73
```

---

## Step 2 — Start the Ray Head Node

Run this on **pc3-4500** (192.168.3.73):

Stop the ray if already running:

```bash
conda activate ray-env
ray stop --force
```

```bash
conda activate ray-env
ray start --head --port=6379 --dashboard-host=0.0.0.0
```

Dashboard is available at `http://192.168.3.73:8265` once the head is up.

---

## Step 3 — Connect Worker Nodes to Head

Run this on **each worker node** (pc1–pc2, pc4–pc8) from a shell where
`.bashrc` has been sourced:

```bash
conda activate ray-env
ray start --address='192.168.3.73:6379'
```

> **Critical:** `GLOO_SOCKET_IFNAME` must be present in the shell that runs
> `ray start`. The Ray worker process inherits env vars only at startup — a
> running worker does not pick up `.bashrc` changes. If you added or changed the
> vars after the worker was already running, restart it:
> ```bash
> ray stop
> ray start --address='192.168.3.73:6379'
> ```
> Do **not** run `ray stop` on the head node (pc3) — it tears down the whole cluster.

Verify all nodes are connected from the head:

```bash
ray status
```

## Step 4 — Verify

```bash
echo "$GLOO_SOCKET_IFNAME $VLLM_HOST_IP"
# RTX 4500 nodes: enp0s31f6 192.168.3.73
# RTX 3090 nodes: eno1 192.168.3.73

which vllm
# expected: /home/user/anaconda3/envs/ray-env/bin/vllm
```

---

## Step 5 — Free the Port + GPUs

If a previous vLLM server is running, stop it before launching a new one:

```bash
sudo systemctl stop vllm 2>/dev/null   # stop the systemd unit if active
pkill -f "vllm serve"                  # stop any manually started instance
sudo fuser -k 8000/tcp 2>/dev/null     # release port 8000 if still bound
sleep 5
ray status | grep GPU                  # should show: 0.0/8.0 GPU
```

> If it still shows `8.0/8.0 GPU` reserved, wait ~10 s and re-run
> `ray status | grep GPU` until it drops to `0.0/8.0`.

---

## Step 6 — Launch vLLM from pc3-4500 (head node)

The driver (rank 0) reads `GLOO_SOCKET_IFNAME` from its own shell. Remote
workers read it from their Ray process environment (set at `ray start` time).
Both must be set correctly — see Steps 1 and 3 above.

```bash
RAY_ADDRESS=192.168.3.73:6379 VLLM_PLUGINS="" \
vllm serve ~/projects/vllm-deployment/vllm/models/3.1-8b-instruct \
  --dtype float16 \
  --tensor-parallel-size 1 \
  --pipeline-parallel-size 8 \
  --distributed-executor-backend ray \
  --gpu-memory-utilization 0.4 \
  --max-model-len 1024 \
  --max-num-seqs 4 \
  --enforce-eager \
  --host 0.0.0.0 \
  --port 8000
```

Server is ready when you see:

```
INFO: Uvicorn running on http://0.0.0.0:8000 (Press CTRL+C to quit)
```

## Troubleshooting

### `NCCL error: unhandled cuda error` on one worker node

**Symptom:** vLLM starts cleanly on all workers except one, which fails at
`ncclCommInitRank` with `RuntimeError: NCCL error: unhandled cuda error`. A
direct two-node PyTorch NCCL test between that node and the head passes — so
the GPU and network are fine.

**Root cause:** The `nvidia-nccl-cu12` pip package can ship different binaries
across installations despite reporting the same version string. A corrupt or
mismatched `libnccl.so.2` binary causes `ncclCommInitRank` to fail only in the
multi-node vLLM context (vLLM loads `libnccl.so.2` via ctypes directly, while
`torch.distributed` uses PyTorch's own bundled NCCL).

Verify with:
```bash
md5sum /home/user/anaconda3/envs/ray-env/lib/python3.12/site-packages/nvidia/nccl/lib/libnccl.so.2
# compare against a working node — hashes must match
```

**Fix:** Copy `libnccl.so.2` from a working node to the failing node:

```bash
# On the failing node — pull from a working worker (e.g. pc7, 192.168.3.77)
scp user@192.168.3.77:/home/user/anaconda3/envs/ray-env/lib/python3.12/site-packages/nvidia/nccl/lib/libnccl.so.2 \
    /home/user/anaconda3/envs/ray-env/lib/python3.12/site-packages/nvidia/nccl/lib/libnccl.so.2
```

Restart the Ray worker and retry. No env var changes needed — the in-place
replacement is picked up automatically.

