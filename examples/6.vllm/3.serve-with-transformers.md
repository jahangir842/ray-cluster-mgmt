# Serving Llama 3.1-8B with Hugging Face Transformers

Serves the same model as the vLLM guides but using the Hugging Face
`transformers` library directly — no vLLM required. Two options are covered:

1. **Quick inference** — a Python script for one-off generation
2. **API server** — a FastAPI server with an OpenAI-compatible `/v1/chat/completions` endpoint

> For production serving with better throughput and continuous batching, prefer
> the vLLM approach in [`2.vllm-on-single-node.md`](2.vllm-on-single-node.md).

---

## Prerequisites

```bash
conda activate ray-env
pip install transformers accelerate fastapi uvicorn
```

---

## Memory budget

Same model, same VRAM requirement as the vLLM single-node guide:

| Item | Size |
|------|------|
| Model weights (float16) | ~15.0 GB |
| Activations / KV cache (runtime) | ~1–3 GB |
| **Minimum GPU** | **24 GB** |

Use `torch_dtype=torch.float16` and `device_map="auto"` — `auto` places the
full model on the GPU if it fits, otherwise it spills layers to CPU (slow).

## Comparison: Transformers vs vLLM

| Feature | Transformers | vLLM |
|---------|-------------|------|
| Setup complexity | Low | Low |
| Throughput (requests/s) | Low — processes one request at a time | High — continuous batching |
| Latency (first token) | Higher | Lower |
| Streaming support | Manual (`TextIteratorStreamer`) | Built-in |
| Multi-GPU (single node) | `device_map="auto"` tensor split | `--tensor-parallel-size N` |
| Multi-node | Not supported natively | `--pipeline-parallel-size N` with Ray |
| OpenAI API compatibility | Manual (as above) | Built-in |

Use transformers for quick experiments and scripting. Use vLLM when serving
multiple users or needing consistent latency.

## Troubleshooting

### `CUDA out of memory` during model load

Another process is holding GPU memory. Check with `nvidia-smi` and kill stale
processes before loading the model.

### Model generates repetitive or empty output

Add `eos_token_id` and `pad_token_id` to `model.generate()`:

```python
output_ids = model.generate(
    input_ids,
    max_new_tokens=200,
    do_sample=False,
    eos_token_id=tokenizer.eos_token_id,
    pad_token_id=tokenizer.eos_token_id,
)
```

### Slow generation (>10 s/token)

`device_map="auto"` may have placed some layers on CPU. Verify the full model
is on GPU:

```python
print({name: param.device for name, param in model.named_parameters()})
```

If any layer shows `cpu`, the GPU lacks free VRAM — free memory or switch to
a quantized model (`load_in_4bit=True` with `bitsandbytes`).
